# Study Note Organizer —— Sunny 精读学习笔记生成器

把一份原始学科资料，转写成 Sunny 习惯的那种**高密度、强提纲、带批判旁注**的精读笔记，并以 Obsidian Markdown 形式落盘。

核心是一条**循环流水线**：不要一口气把整份资料糊成一篇笔记，而是**切成若干部分，每部分单独「整理 → 自检 → 修正 → 放行」，确认无误再做下一部分**。这样每一段都经过检查，避免长资料后半截质量滑坡或漏掉关键数字。

---

## 为什么这样设计

Sunny 记笔记不是抄原文，而是**重构原文的逻辑骨架**：抓主线、列方法、扒数字、挑毛病。资料越长越容易在后半段偷懒或丢信息，所以用**逐部分闭环**的方式强制每段都达标。整理风格的灵魂是「我读完这一节，脑子里留下的结构是什么」——你要还原的是那个结构，不是段落本身。

详细的风格示范见 `references/style-guide.md`，写每一部分前**先读它**，对照着写。

## 工作流程（循环）

### 第 0 步：通读 + 切分（只做一次）

1. 先把整份资料**快速通读一遍**，掌握全貌：它是什么类型（论文 / 教材章节 / PPT / 综述）、主线是什么。
2. 按资料**自身的天然结构**切分成「部分」。优先级：
   - 论文 → 按 摘要 / 介绍 / Section 2 / Section 3 / … 切；
   - 教材 → 按 节（§2.1、§2.2）切；
   - PPT / 讲义 → 按主题块切。
3. 每个「部分」应是一个能独立消化的单元（通常 1 节 / 一个 Section）。把切分结果列出来，作为循环的 checklist。
4. 在输出文件顶部先写好 **frontmatter + 一级标题 + 目录占位**（见下方输出规范），然后进入循环。

### 第 1～N 步：逐部分循环

对**每一个**部分，依次执行 `整理 → 自检 → 修正 → 放行`，**当前部分自检通过后，才能开始下一部分**：

**(a) 整理**　按 `references/style-guide.md` 的风格，把这一部分写成笔记，追加到输出文件。重点动作：
   - 还原**层级提纲**（见下方「风格速查」）；
   - **理工科知识点按下方 `## 理工科知识点模板` 展开**——定义 / 性质 / 公式及推导 / 图 / 物理与数学核心，缺一不可；
   - 把关键结论用 `==高亮==`，专业术语和**所有数字/公式量**用 `**加粗**`；
   - 凡是涉及「机理 / 数据流 / 流程 / 结构布局 / 几何关系」的地方，画一个 **ASCII 示意图**；
   - 每个方法 / 定理 / 公式后面，补一条**批判旁注**：成立条件、适用边界、易错点、缺陷是什么；
   - 自己的疑问、吐槽、存疑点，用 `（……）` 或 `> [!question]` 就地标注；
   - **达到本节字数下限**（见模板末尾「字数限定」），知识点展开要充分，不能用一句话打发。

**(b) 自检**　对照下方 `## 自检清单` 逐条核对当前部分。**不要跳过这一步**——这是循环的意义所在。

**(c) 修正**　哪条没达标就补哪条（漏了数字补数字，漏了缺陷补缺陷，散文化了就改回提纲）。

**(d) 放行**　在 checklist 上勾掉这一部分，**简短**告诉用户「第 X 部分（节名）已整理并自检通过」，然后进入下一部分。不要长篇复述内容，用户能自己看文件。

### 收尾步（全部部分完成后）

1. 回到文件顶部，把**目录**补全（链接到各节标题）。
2. 在文末加一节 `## 速记卡 / 一句话总结`：用 5～10 条把整份资料的核心结论浓缩成「考前能扫一眼」的清单（Sunny 喜欢把关键数字集中放这里）。
3. 整份做一次**总检**：术语前后是否一致、数字是否和原文一致、有没有断裂的逻辑链。
4. 文件存到 Obsidian 库里（默认与原资料**同目录**，文件名 `<原资料名>_精读笔记.md`；除非用户另指定路径）。

## 分级叙事结构（硬性要求——知识骨架的命脉）

笔记好不好用，**第一眼看的是结构**。Sunny 的笔记必须用**固定的多级序号体系逐级叙事**，让任何一节的逻辑层次一眼可辨。这是硬规则，不是建议。

**固定的五级骨架，从外到内严格对应（注意缩进用「每级 2 空格」，见下方「缩进与渲染」硬性要求）：**

```
##            —— 顶层主题（摘要 / 介绍 / Section 2 / §2.1 …），章节级
一·二·三·      —— 中文序号：大类 / 主线分支            （0 空格）
  1. 2. 3.    —— 阿拉伯数字：该大类下的要点            （2 空格）
    （1）（2） —— 括号数字：要点的细分 / 步骤 / 分项     （4 空格）
      - …     —— 短横线：最细的并列项、旁注、缺陷       （6 空格）
```

**逐级叙事的规则（保证有条理）：**

1. **逐级不跳级**：`一·` 下面才是 `1.`，`1.` 下面才是 `（1）`。不能从 `一·` 直接跳到 `（1）`，也不能在没有 `一·` 的情况下凭空出现 `1.`。
2. **同级即并列**：同一层的条目必须是**逻辑平行**的兄弟项（都是「方法」，或都是「步骤」，或都是「性质」），不能一个是方法、一个是它的缺点混在同级。
3. **父级统摄子级**：子级必须是父级的**展开 / 下属 / 细化**，读「父→子」要能顺下来。
4. **一条一义**：每个序号项只承载一个意思，长了就往下分级，别把三件事塞进一行。
5. **序号符号统一**：自始至终用 `一·二·三· / 1.2.3. / （1）（2）（3） / -` 这一套，**不要中途换成 a/b/c 或 ①②③ 或 1)2)3)**，全文一致才有条理感。
6. **推导步骤也分级**：公式推导的多步，用 `（1）（2）（3）` 或独立编号承载，每步一行、标依据——分级本身就是把逻辑摊开。
7. **五件套也必须挂在序号上（最易被忽视，重点强调）**：每个知识点内部的「定义 / 性质 / 公式与推导 / 图 / 核心理解」**不是散落的加粗段落**，而要**逐项编号成 `一·二·三·四·五·`**——即 `一·定义`、`二·性质`、`三·公式与推导`、`四·图`、`五·核心理解`。它们的下属要点顺势降一级用 `1.2.3.`，再细分用 `（1）（2）（3）`，最细用 `-`。**绝不允许出现「裸的加粗标签 + 无编号正文」**，否则知识点内部就没有骨架了。

> [!note] 为什么死磕分级
> 分级序号不是排版洁癖，它是**强制你把知识的从属与并列关系想清楚**。能干净地分进 `一·→1.→（1）`，说明你真的理清了结构；分不进去，往往是自己没读透。结构清晰 = 理解到位的副产品。

**反例（别这么写）：**
```
压气机失稳：攻角过大会失速，方法有筛查变量但只适用不良流场，还有逐级优化
```
**正例（分级叙事）：**
```
一·失稳机理
    1.攻角过大（低转速/流量偏小）→ 叶背分离 → 失速
二·解决方法及缺陷
    1.筛查变量——仅适用于明显不良流场
    2.逐级优化——忽略前后级相互影响
```

## 缩进与渲染（Obsidian 兼容性——硬性！直接决定公式能否显示）

> 这条不是排版细节，它决定笔记里的 `$$` 公式会不会变成一坨灰底乱码。务必照做。

**根因**：标准 Markdown（CommonMark）把**行首 ≥4 个空格**的行解析为**缩进代码块**（灰底方框）。代码块里"所见即所得"——`$$...$$`、`$...$` 被当纯文本原样显示，**不进 MathJax 渲染**。于是「4 空格起首的层级行」既触发灰框、又让公式失效。

**所以分级缩进一律「每级 2 空格」，不是 4：**

```
一·       → 0 空格
  1.      → 2 空格   ← <4，是普通段落，不触发代码块
    （1）  → 4 空格   ← 紧随上一行的“段落续行”（lazy continuation），不独立成代码块
      -   → 6 空格   ← 续行
```

只要**首级（`1.`）≤3 空格**（用 2），首级就是普通段落；后面更深的缩进只是同一段落的续行，被首级"吞"进去，`$$` 公式照常渲染。

**两条配套硬规则：**

1. **每级递进 2 空格**（0 / 2 / 4 / 6 …），**绝不用 4 空格或 Tab 起首级**（Tab 通常等价 4 空格，同样触发代码块）。
2. **同一编号组内部不要插空行**：`1.` 与其下属 `（1）（2）-` 之间不空行，让深缩进作为"段落续行"被吞掉（一旦中间空行，后面的 4/6 空格行又会被当成新的代码块）。确实需要空行分隔时，改用真正的 Markdown 列表 `-` 承载，别留**裸的 ≥4 空格缩进行**。

> [!warning] 唯一例外
> 你**有意**画的 ASCII 图、写的公式块，用的是 ` ``` ` 三反引号**围栏代码块**——那是故意的、不受影响。本规则只针对"无意"被缩进触发的代码块。

## 风格速查（写每一部分时对照）

层级提纲的缩进骨架（详见上方「分级叙事结构」+「缩进与渲染」，每级 2 空格、首级 <4，是硬性要求）：

```
顶层主题（摘要 / 介绍 / Section 2 …）   —— 用 ## 标题
一·二·三·                              —— 中文序号，大类     （0 空格）
  1. 2. 3.                             —— 阿拉伯数字，要点   （2 空格）
    （1）（2）（3）                      —— 括号数字，细分     （4 空格）
      - 更细的并列项 / 旁注              —— 短横线，最深层     （6 空格）
```

- **不要写成散文段落。** Sunny 的笔记是**提纲 + 短句**，省略主语、信息密度高。一句话能讲清就不写两句。
- `→` 表示因果 / 转变 / 推导（如「转速下降 → 速度三角形改变 → 攻角过大 → 失速」）。
- `==高亮==` 只给**最关键的结论**（如「==减少参数维度数——最有前景==」），不要滥用。
- `**加粗**` 给专业术语首次出现、以及**每一个性能数字**（效率 +0.48%、马赫 1.4、约 48 小时、25 个变量）。数字是精读笔记的命根子，绝不能丢。
- **批判性**是这套笔记的标志：方法讲完必列缺陷。例：「逐级优化 → 忽略了前后级的相互影响」。
- 善用**类比**降低理解门槛：「转子叶片本质上是排机翼」。
- Obsidian 专属语法可用：`[[文件名]]` 链接同目录资料、`$...$` 写公式、`> [!note]` / `> [!question]` callout 放旁注。

### ASCII 示意图——什么时候画、怎么画

只要内容是「输入怎么变成输出」「结构怎么排布」「流程分几步」，就画图。两种常用模板：

数据流 / 机理（竖向箭头）：
```
16 个控制点 (输入)
      │
      ▼
  贝塞尔面公式 = 加权混合
      │
      ▼
叶片每点位移 Δ (输出) →  原叶片 + Δ = 新叶片
```

结构布局（横向排布）：
```
S0(IGV) | R1 S1 | R2 S2 | R3 S3 | R4 S4 S5
进口导叶  ←———— 4 个级 ————→
```

## 理工科知识点模板

理工科笔记的价值在于**把一个知识点讲透**，不是列个名字就过。每遇到一个核心知识点（定理、公式、模型、概念），按下面这套「五件套」展开——**有则写，没有则跳过该项，但定义、核心理解两项几乎总要有**：

### 五件套

1. **定义**　把概念**完整、严格**地写出来，不能只给名字。讲清楚它是什么、研究对象是什么、关键术语指什么。
2. **性质 / 定理**　关键性质、定理陈述、**成立条件**、适用范围。条件是理工科的命门，必须写明（如「仅在无粘、定常假设下成立」）。
3. **公式及推导**　这是精读笔记**最不可替代**的部分，绝不能只甩结论。
   - 公式用 Obsidian 数学语法：行内 `$E=mc^2$`，独立成行 `$$ ... $$`；
   - **推导必须分步、不跳步**：从起点（某守恒律 / 定义 / 已知式）出发，**每一步单独成行 + 标依据**（用了什么定理 / 假设 / 近似 / 几何关系），直到目标式。原文写「易证」「显然」的，正是你要补全的地方。
   - **只甩一个 `$$结果$$` 不带过程 = 自检不通过**。哪怕原文没给，也用知识库补出标准推导（并按补充规则标注）。
   - 推导步骤用 `（1）（2）（3）` 承载，每步一行。
   - **范本——离心拉伸应力（叶根）从微元推到积分式：**
     ```markdown
     **公式与推导**：求任意截面 Zⱼ 的离心拉伸应力 σ_c。
     （1）取叶高 Z 处微元段 dZ，其质量 $dm=\rho A(Z)\,dZ$        ← 密度×微元体积
     （2）该微元产生的离心力 $dP=dm\cdot\omega^2 R(Z)=\rho\omega^2 R(Z)A(Z)\,dZ$   ← 离心力定义 F=mω²R
     （3）Zⱼ 截面承受其**以上全部**微元的离心力之和（叶尖自由、叶根固支）：
         $$P(Z_j)=\int_{Z_j}^{H}\rho\,\omega^2 R(Z)A(Z)\,dZ$$       ← 沿叶高积分到叶尖 H
     （4）除以该截面面积得平均拉应力：
         $$\sigma_c(Z_j)=\frac{P(Z_j)}{A(Z_j)}=\frac{1}{A(Z_j)}\int_{Z_j}^{H}\rho\,\omega^2 R(Z)A(Z)\,dZ$$   ← 应力=力/面积
     → 推论：Zⱼ 越小（越靠叶根）积分上限越大 → **叶根应力最大（危险截面）**；叶尖 σ_c=0。
     ```
     注意每步右侧的 `← 依据`——这就是「不跳步」的样子：读者照着能自己复算。
4. **图**　分两种情形处理，目标是让 Sunny 能**手动补真图**：
   - **简单示意（数据流 / 流程 / 结构排布）**→ 直接画 **ASCII 图**（模板见上方「ASCII 示意图」），轻量、够用。
   - **复杂几何 / 矢量图（速度三角形、受力图、叶栅几何、曲线坐标图、电路）**→ ASCII 画不清楚也容易乱，**改为留「手动插图位」**：插入 Obsidian 图片占位 + 图注，让 Sunny 之后用 Excalidraw / 截图 / TikZ 自己补真图。格式：
     ```markdown
     ![[图_速度三角形.excalidraw]]   <!-- 📌 待补图 -->
     > **图注**：基元级速度三角形。需画出 C₁（绝对速度）、W₁（相对速度）、U（轮缘速度）的矢量合成关系，标注气流角 β₁。
     ```
     占位统一加 `<!-- 📌 待补图 -->` 注释，方便 Sunny 全局搜 `📌` 一次补齐所有图。
   - 不论哪种，都要**先用文字把图的关键信息说清**（坐标轴、趋势、拐点、物理含义），这样即使图没补，笔记也能读懂。
   - 判断准则：**ASCII 能画清就画 ASCII；画出来会糊/会歧义，就留插图位**，别硬用字符拼复杂矢量图。
5. **物理 & 数学核心**　这是精读笔记的灵魂——点透「**为什么是这样**」：
   - **物理直觉**：这个公式/现象背后的物理图像是什么（一两句大白话，可用类比）；
   - **数学本质**：它在数学上是什么结构（守恒律？极值问题？线性变换？特征值？），为什么这么列式。
   - 用 `> [!note] 核心理解` callout 单独框出来，方便复习时一眼抓重点。

### 套用示例（骨架）

**五件套一律挂在 `一·二·三·四·五·` 上**（见「分级叙事结构」规则 7），不要写成裸的加粗段落：

```markdown
### <知识点名称>

一·**定义**：……（完整严格）

二·**性质 / 条件**：
  1. ……（成立条件、适用范围）
  2. ……

三·**公式与推导**：
$$ ... $$
  1. …… ← 依据/物理意义
  2. …… ← ……
  → 结论：……

四·**图**：简单示意用 ASCII；复杂矢量图留插图位：
`![[图_xxx.excalidraw]]` <!-- 📌 待补图 --> + `> **图注**：……`（先文字说清关键信息）

五·**核心理解**（物理 + 数学）：
> [!note] 核心理解
> 物理上：……；数学上：……

六·**易错 / 边界**：……（可选，作为第六项续编号）
```

### 字数限定（不能总结得太简单）

防止把知识点压成干瘪一行。默认下限，**随原材料体量上下浮动**（原文这一节越厚，下限越高）：

- **核心章 / 重点小节**：正文 **≥ 600～800 字**，且每个核心知识点五件套基本齐全；
- **一般小节**：**≥ 300 字**；
- **纯过渡 / 列举性内容**：可适当放宽，但仍要把要点写全。

字数是**手段不是目的**：宁可详尽不可过简，但也别灌水凑字。判断标准是「**这一节读完，能不能独立复习、能不能照着把公式推一遍**」——能，就够了；不能，就是太简，回去补定义、补推导、补核心理解。

---

## 自检清单（每部分必过）

逐条核对当前这一部分，没过就回去补。**标⚠️的是硬性项——不通过绝不放行**：

1. **分级叙事对不对**（重点查）：是否严格用 `一·二·三· → 1.2.3. → （1）（2）（3） → -` 逐级展开？没跳级？同级都是并列项？符号全文统一？知识的主次/从属一眼可辨？
   - ⚠️**缩进合规（硬卡，关乎公式渲染）**：层级缩进是否**每级 2 空格、首级 ≤3 空格**？有没有哪一行因 ≥4 空格起首被 Obsidian 渲染成灰底代码块、把 `$$` 公式废掉？编号组内部有没有误插空行导致深缩进行又变代码块？
2. **五件套齐不齐 + 是否编号**（重点查）：每个核心知识点是否覆盖了 定义 / 性质条件 / 公式推导 / 图 / 物理数学核心？该有的没缺？⚠️并且这五件套是否**都挂在 `一·二·三·四·五·` 上**（规则 7），而不是写成裸的加粗段落「**定义**：… **性质**：…」？出现无编号的浮动加粗标签 → 不通过，回去补编号。
3. **定义完整**：核心概念是否给了**严格完整**的定义，而不是只提了名字？
4. ⚠️**推导分步标依据**（最易偷懒，硬卡）：凡出现公式，是否**分步推导 + 每步标依据**，而不是只甩一个 `$$结果$$`？读者能照着复算吗？只有结论没过程 → 不通过，回去补推导（原文没给就用知识库补并标注）。
5. **核心理解到位**：是否点透了「为什么是这样」（物理直觉 **+ 数学本质**）？数学本质那半句别省。每章至少 1 个 `> [!note] 核心理解`。
6. **字数达标**：本节是否达到字数下限？有没有把核心知识点压成干瘪一两行？太简就回去展开。
7. **数字/公式全不全**：原文这一部分的关键数字、参数、百分比、公式，是否**全部**抓进来并加粗/用 `$$`？（最容易漏）
8. **结论高亮**：本部分最关键的一两个结论是否 `==高亮==`？
9. **边界/易错批注**：每个方法/定理/公式是否补了成立条件、适用边界或易错点？
10. **图处理对不对**：简单示意画了 ASCII？复杂几何/矢量图（速度三角形、受力图等）是否**留了 `![[...]]` 手动插图位 + 图注 + `📌` 标记**，而不是硬拼成乱码 ASCII？图的关键信息有没有先用文字说清？
11. **术语一致**：专业术语和前面部分用词是否统一？
12. **忠实原文**：有没有把原文/公式记反或记错？存疑处是否用「（？）」标了出来而不是瞎编？
13. ⚠️**补充必带标注**（硬卡）：用自身知识补的内容（前置定义、补全的推导、新闻数据、工程经验等），是否**无一例外**都用 `> [!tip] 补充` 或 `（补：…）` 和原文分开？补的是否可靠（没编公式/数字）？凡来自模型而非原文的，没标 = 不通过。

全过（尤其两个 ⚠️）→ 放行，进入下一部分。

## 知识库补充（用 agent 自身知识补全原文）

原资料常常**默认前置知识、跳过标准推导、不给背景**。Sunny 的精读笔记要好用，就该把这些缺口补上——**你（agent）可以动用自身知识库主动补充适当内容**，让笔记自洽、能独立复习。

**什么时候补：**
- 原文用到却没给定义的**前置概念**（如提到「雷诺数」却没解释）→ 补一句严格定义；
- 原文**跳过的标准推导**（「易证」「显然」）→ 把中间步骤补出来；
- 公式/定理缺**成立条件、量纲、典型取值范围**→ 补上；
- 缺**物理直觉 / 数学本质** → 用你的理解点透（这正是核心理解项）；
- 一个常见但原文没提的**易错点 / 反例 / 工程经验** → 补一条。

**铁律——补充必须可区分、可信：**
- **标注来源**：补充内容统一用 `> [!tip] 补充` callout，或行内打 `（补：……）`，**和原文内容明确分开**，让 Sunny 一眼知道哪些是原文、哪些是你加的。
- **只补成熟可靠的知识**：教科书级、共识性的内容才补；前沿/有争议/记不准的，**标存疑**（`（补，存疑：……）`）或干脆不补，绝不编造公式和数字。
- **不喧宾夺主**：补充是补缺口，不是另写一篇；服务于理解原文，别跑题。
- **不改写原文结论**：补充与原文冲突时，不直接覆盖，而是并列指出（`> [!warning] 此处与原文似有出入：……`）。

示例：
```markdown
**雷诺数 Re**
（补）原文直接用了 Re 未定义。

> [!tip] 补充
> 定义：$Re = \dfrac{\rho v L}{\mu}$，惯性力与粘性力之比。
> 物理意义：Re 小 → 粘性主导、层流；Re 大 → 惯性主导、湍流。
> 工程经验：管内流动临界 $Re \approx 2300$。
```

## token 经济性（省 token，但不掉质量）

把算力花在「读资料 + 写好笔记」这件正事上，别浪费在无谓的重复读取。下面几条**只砍浪费、不砍深度**：

1. **reference 只读一次**：`references/style-guide.md` 在开始循环前读一遍即可，**别每节都重读**——风格已在上下文里。
2. **自检只针对当前节**：自检对照的是**刚写完、还在上下文里的这一节**，**不要回读整个越来越长的输出文件**（那会随章节数平方级烧 token）。需要跨节核对术语一致性时，只回看相关的几行，别整文件重扫。
3. **原资料按需读**：长 PDF/资料**按部分读取**（用 offset/limit 或按页），处理完一部分再读下一部分，不要一次性全量灌进上下文又反复重读。
4. **批次别太碎**：切分部分时，一个能独立消化的单元（约一节）为一批；过度细碎会让循环开销（自检、汇报）摊薄到太多次。
5. **汇报极简**：每节完成一句话带过，内容留文件里，**绝不在对话里复述笔记正文**。
6. **长资料可两段式**（可选，征求用户意见）：资料特别长时，可先全篇产**骨架版**（提纲 + 每节核心结论），再按 Sunny 指定的重点章逐节展开五件套——把深度 token 花在真正要精读的地方。

判断原则：**省的是"重复读取/复述/回扫"，留的是"定义、推导、核心理解"**。质量项一个都不能因为省 token 而砍。

## 几条铁律

- **分级叙事是硬骨架**。一·→1.→（1）→- 逐级不跳级、同级并列、符号统一；结构清晰是理解到位的体现。
- ⚠️**缩进每级 2 空格、首级 <4**。≥4 空格起首会被当成代码块，`$$` 公式直接不渲染——这是低级但致命的错。组内不要插空行。
- **逐部分闭环，别图快**。整完一节先自检再下一节；用户要的就是「每部分都检查过」的踏实感。
- **理工科讲透，别压缩成一行**。定义、推导、核心理解三者缺一不可；达不到字数下限就是没展开够。
- **推导不跳步**。公式要能让读者照着复算，关键步骤标依据。
- **数字/公式一个都不能丢**。精读笔记里数字与推导 > 形容词。
- **保留批判视角**。这不是摘要，是带判断的精读——方法的毛病、公式的成立条件要写出来。
- **简洁汇报**。每部分完成后一句话带过，把内容留在文件里，别在对话里复述。
- **可补充，但要标注**。用自身知识补全原文缺口时，`> [!tip] 补充` 区分开，只补可靠知识，不编造。
- **省 token 不掉质量**。砍重复读取/复述/回扫，留定义/推导/核心理解。
- 不确定原文意思时，**标注存疑**而不是编造。
