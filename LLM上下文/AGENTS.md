## 4. 整理工作流（循环流水线）

把一份原始资料转写成精读笔记，**不要一口气糊成整篇**，而是切成若干部分逐段闭环。

### 第 0 步：通读 + 切分（只做一次）

1. 快速通读全文，判类型（论文 / 教材章节 / PPT / 综述）与主线。
2. 按资料自身的天然结构切「部分」：论文按摘要 / 介绍 / Section 切；教材按节
   （§2.1、§2.2）切；PPT 按主题块切。每部分应是一个能独立消化的单元。
3. 列出切分结果作为循环 checklist。
4. 在输出文件顶部先写好 **frontmatter + 一级标题 + 目录占位**，再进入循环。

### 第 1～N 步：逐部分循环

对每个部分依次执行 `整理 → 自检 → 修正 → 放行`，**当前部分自检通过后才能进入
下一部分**：

- **(a) 整理**：按 §6 风格与 §5 五件套模板，把这一部分写成笔记追加到输出文件。
- **(b) 自检**：对照 §7 的 13 条自检清单逐条核对当前部分。**不要跳过**。
- **(c) 修正**：哪条没达标补哪条。
- **(d) 放行**：在 checklist 上勾掉，**一句话**告诉用户「第 X 部分已整理并自检
  通过」，不在对话里复述内容。

### 收尾步

1. 回顶部补全**目录**（链接各节标题）。
2. 文末加 `## 速记卡 / 一句话总结`：5～10 条把核心结论浓缩成「考前能扫一眼」的
   清单（Sunny 喜欢把关键数字集中放这里）。
3. 整份总检：术语前后是否一致、数字是否和原文一致、有没有断裂的逻辑链。
4. 落盘：默认与原资料**同目录**，文件名 `<原资料名>_精读笔记.md`（除非用户另指
  定路径）。

### token 经济性（省 token 不掉质量）

把算力花在「读资料 + 写好笔记」上，别浪费在无谓重复读取：

1. 风格指南只读一次（开始循环前），别每节重读。
2. 自检只针对当前节（还在上下文里），**不要回读整个越来越长的输出文件**。
3. 原资料按部分读取（长 PDF 按页 / 按节），处理完一部分再读下一部分。
4. 切分别太碎——一节能独立消化为一批。
5. 汇报极简，每节一句话带过，**绝不在对话里复述笔记正文**。
6. 资料特别长时可两段式（先征求用户意见）：先产骨架版（提纲 + 每节核心结论），
   再按重点章逐节展开五件套。

判断原则：**省的是「重复读取 / 复述 / 回扫」，留的是「定义、推导、核心理解」**。
质量项一个都不能因省 token 而砍。

## 5. 精读五件套模板（理工科知识点硬标准）

每个核心知识点（定理、公式、模型、概念）按「五件套」展开——**有则写，没有则
跳过该项，但定义、核心理解两项几乎总要有**。五件套**必须挂在 `一·二·三·四·五·`
编号上**（见 §6 分级叙事规则 7），不允许写成裸的加粗段落。

### 五件套

1. **定义**　把概念**完整、严格**地写出来，不能只给名字。讲清它是什么、研究对象
   是什么、关键术语指什么。
2. **性质 / 定理**　关键性质、定理陈述、**成立条件**、适用范围。条件是理工科的命
   门，必须写明（如「仅在无粘、定常假设下成立」）。
3. **公式及推导**　精读笔记**最不可替代**的部分，绝不能只甩结论。
   - 公式用 Obsidian 数学语法：行内 `$E=mc^2$`，独立成行 `$$ ... $$`。
   - **推导必须分步、不跳步**：从起点（守恒律 / 定义 / 已知式）出发，每步单独成
     行 + 标依据（用了什么定理 / 假设 / 近似 / 几何关系），直到目标式。原文写
     「易证」「显然」的，正是要补全的地方。
   - **只甩一个 `$$结果$$` 不带过程 = 自检不通过**。原文没给就用知识库补出标准
     推导并按 §8 标注规则标注。
   - 推导步骤用 `（1）（2）（3）` 承载，每步一行。
4. **图**　分两种情形：
   - **简单示意（数据流 / 流程 / 结构排布）** → 直接画 ASCII 图（模板见 §6）。
   - **复杂几何 / 矢量图（速度三角形、受力图、叶栅几何、电路）** → ASCII 画不清
     就留「手动插图位」：插入 Obsidian 图片占位 + 图注，让 Sunny 之后用 Excalidraw
     / 截图 / TikZ 自己补真图。格式：
     ```markdown
     ![[图_速度三角形.excalidraw]]   <!-- 📌 待补图 -->
     > **图注**：基元级速度三角形。需画出 C₁（绝对速度）、W₁（相对速度）、
     > U（轮缘速度）的矢量合成关系，标注气流角 β₁。
     ```
     占位统一加 `<!-- 📌 待补图 -->` 注释，方便全局搜 `📌` 一次补齐。
   - 不论哪种，都要**先用文字把图的关键信息说清**（坐标轴、趋势、拐点、物理
     含义），即使图没补笔记也能读懂。
   - 判断准则：**ASCII 能画清就画 ASCII；画出来会糊 / 会歧义就留插图位**。
5. **物理 & 数学核心**　精读笔记的灵魂——点透「**为什么是这样**」：
   - 物理直觉：背后的物理图像是什么（一两句大白话，可用类比）。
   - 数学本质：它在数学上是什么结构（守恒律？极值问题？线性变换？特征值？）。
   - 用 `> [!note] 核心理解` callout 单独框出来。

可选第 6 项 `六·**易错 / 边界**`。

### 套用骨架

```markdown
### <知识点名称>

一·**定义**：……（完整严格）

二·**性质 / 条件**：
  1. ……（成立条件、适用范围）
  2. ……

三·**公式与推导**：
$$ ... $$
  1. …… ← 依据 / 物理意义
  2. …… ← ……
  → 结论：……

四·**图**：简单示意用 ASCII；复杂矢量图留插图位：
`![[图_xxx.excalidraw]]` <!-- 📌 待补图 --> + `> **图注**：……`（先文字说清关键信息）

五·**核心理解**（物理 + 数学）：
> [!note] 核心理解
> 物理上：……；数学上：……

六·**易错 / 边界**：……（可选，作为第六项续编号）
```

### 字数下限

防止把知识点压成干瘪一行。**随原材料体量上下浮动**（原文这一节越厚，下限越高）：

- **核心章 / 重点小节**：正文 **≥ 600～800 字**，核心知识点五件套基本齐全。
- **一般小节**：**≥ 300 字**。
- **纯过渡 / 列举性内容**：可适当放宽，但仍要把要点写全。

判断标准：「**这一节读完，能不能独立复习、能不能照着把公式推一遍**」——能就够；
不能就是太简，回去补定义、补推导、补核心理解。

## 6. 分级叙事结构 + 缩进（硬规则）

### 五级骨架

笔记好不好用，第一眼看的是结构。必须用固定的多级序号逐级叙事：

```
##            —— 顶层主题（摘要 / 介绍 / Section 2 / §2.1 …），章节级
一·二·三·      —— 中文序号：大类 / 主线分支            （0 空格）
  1. 2. 3.    —— 阿拉伯数字：该大类下的要点            （2 空格）
    （1）（2） —— 括号数字：要点的细分 / 步骤 / 分项     （4 空格）
      - …     —— 短横线：最细的并列项、旁注、缺陷       （6 空格）
```

逐级叙事规则：

1. **逐级不跳级**：`一·` 下面才是 `1.`，`1.` 下面才是 `（1）`。
2. **同级即并列**：同一层条目逻辑平行（都是「方法」/「步骤」/「性质」），不能混
   异类。
3. **父级统摄子级**：子级是父级的展开 / 下属 / 细化。
4. **一条一义**：每个序号项只承载一个意思，长了就往下分级。
5. **序号符号统一**：自始至终用 `一·二·三· / 1.2.3. / （1）（2）（3） / -`，不要
   中途换成 a/b/c 或 ①②③ 或 1)2)3)。
6. **推导步骤也分级**：公式推导的多步用 `（1）（2）（3）` 承载，每步一行、标依据。
7. **五件套必须挂在序号上（最易被忽视，重点强调）**：知识点内部的「定义 / 性质
   / 公式与推导 / 图 / 核心理解」**逐项编号成 `一·二·三·四·五·`**，下属要点降一
   级用 `1.2.3.`，再细分用 `（1）（2）（3）`，最细用 `-`。**绝不允许出现裸的加粗
   标签 + 无编号正文**。

### 缩进与渲染（Obsidian 兼容性——硬卡！决定公式能否显示）

**根因**：CommonMark 把**行首 ≥4 个空格**的行解析为**缩进代码块**（灰底方框）。
代码块里「所见即所得」——`$$...$$`、`$...$` 被当纯文本，**不进 MathJax 渲染**。
于是「4 空格起首的层级行」既触发灰框、又让公式失效。

**所以分级缩进一律「每级 2 空格」，不是 4：**

```
一·       → 0 空格
  1.      → 2 空格   ← <4，是普通段落，不触发代码块
    （1）  → 4 空格   ← 紧随上一行的"段落续行"，不独立成代码块
      -   → 6 空格   ← 续行
```

只要**首级（`1.`）≤3 空格**（用 2），首级就是普通段落；后面更深的缩进只是同段
落的续行，被首级"吞"进去，`$$` 公式照常渲染。

**两条配套硬规则：**

1. **每级递进 2 空格**（0 / 2 / 4 / 6 …），**绝不用 4 空格或 Tab 起首级**（Tab 通常
   等价 4 空格）。
2. **同一编号组内部不要插空行**：`1.` 与其下属 `（1）（2）-` 之间不空行，让深缩
   进作为"段落续行"被吞掉（一旦中间空行，后面的 4/6 空格行又会被当成新代码块）。
   确实需要空行分隔时，改用真正的 Markdown 列表 `-` 承载。

> [!warning] 唯一例外
> 有意画的 ASCII 图、写的公式块，用 ` ``` ` 三反引号**围栏代码块**——那是故意的、
> 不受影响。本规则只针对"无意"被缩进触发的代码块。

### 标注约定（高亮 / 加粗 / 箭头）

- `==高亮==`：只给**最关键的结论**（一节顶多 1～2 处），如「==减少参数维度数——
  最有前景==」，不要滥用。
- `**加粗**`：专业术语**首次出现**、以及**每一个性能数字**（效率 +0.48%、马赫 1.4、
  约 48 小时、25 个变量）。数字是精读笔记的命根子，绝不能丢。
- `→`：因果 / 转变 / 推导（如「转速下降 → 速度三角形改变 → 攻角过大 → 失速」）。
- 善用**类比**降低理解门槛：「转子叶片本质上是排机翼」。
- Obsidian 专属语法：`[[文件名]]` 链同库资料、`$...$` 写公式、`> [!note]` /
  `> [!question]` callout 放旁注。

## 7. 自检清单（每部分必过，13 条）

逐条核对当前这一部分，没过就回去补。**标 ⚠️ 的是硬性项——不通过绝不放行**：

1. **分级叙事对不对**：是否严格用 `一·二·三· → 1.2.3. → （1）（2）（3） → -`
   逐级展开？没跳级？同级并列？符号全文统一？
   - ⚠️**缩进合规（硬卡，关乎公式渲染）**：层级缩进是否**每级 2 空格、首级
     ≤3 空格**？有没有哪一行因 ≥4 空格起首被 Obsidian 渲染成灰底代码块、把 `$$`
     公式废掉？编号组内部有没有误插空行导致深缩进行又变代码块？
2. **五件套齐不齐 + 是否编号**：每个核心知识点是否覆盖了 定义 / 性质条件 / 公式
   推导 / 图 / 物理数学核心？⚠️并且这五件套是否**都挂在 `一·二·三·四·五·`**
   （规则 7），而不是写成裸的加粗段落「**定义**：… **性质**：…」？出现无编号的
   浮动加粗标签 → 不通过，回去补编号。
3. **定义完整**：核心概念是否给了**严格完整**的定义，而不是只提了名字？
4. ⚠️**推导分步标依据**（最易偷懒，硬卡）：凡出现公式，是否**分步推导 + 每步标
   依据**，而不是只甩一个 `$$结果$$`？读者能照着复算吗？只有结论没过程 → 不
   通过，回去补推导（原文没给就用知识库补并标注）。
5. **核心理解到位**：是否点透了「为什么是这样」（物理直觉 **+ 数学本质**）？数学
   本质那半句别省。每章至少 1 个 `> [!note] 核心理解`。
6. **字数达标**：本节是否达到字数下限？有没有把核心知识点压成干瘪一两行？
7. **数字 / 公式全不全**：原文这一部分的关键数字、参数、百分比、公式是否**全部**
   抓进来并加粗 / 用 `$$`？（最容易漏）
8. **结论高亮**：本部分最关键的一两个结论是否 `==高亮==`？
9. **边界 / 易错批注**：每个方法 / 定理 / 公式是否补了成立条件、适用边界或易错点？
10. **图处理对不对**：简单示意画了 ASCII？复杂几何 / 矢量图是否**留了 `![[...]]`
    手动插图位 + 图注 + `📌` 标记**，而不是硬拼成乱码 ASCII？图的关键信息有没有
    先用文字说清？
11. **术语一致**：专业术语和前面部分用词是否统一？
12. **忠实原文**：有没有把原文 / 公式记反或记错？存疑处是否用「（？）」标了出来
    而不是瞎编？
13. ⚠️**补充必带标注**（硬卡）：用自身知识补的内容（前置定义、补全的推导、工程
    经验等），是否**无一例外**都用 `> [!tip] 补充` 或 `（补：…）` 和原文分开？补
    的是否可靠（没编公式 / 数字）？凡来自模型而非原文的，没标 = 不通过。

全过（尤其三个 ⚠️）→ 放行，进入下一部分。

## 8. 知识库补充（用自身知识补全原文）

原资料常默认前置知识、跳过标准推导、不给背景。Sunny 的精读笔记要能用，就该把
这些缺口补上——agent 可以动用自身知识**主动补充适当内容**让笔记自洽。

**什么时候补：**
- 原文用到却没给定义的**前置概念**（如提到「雷诺数」却没解释）→ 补一句严格定义。
- 原文**跳过的标准推导**（「易证」「显然」）→ 把中间步骤补出来。
- 公式 / 定理缺**成立条件、量纲、典型取值范围** → 补上。
- 缺**物理直觉 / 数学本质** → 用自己的理解点透（这正是核心理解项）。
- 一个常见但原文没提的**易错点 / 反例 / 工程经验** → 补一条。

**铁律——补充必须可区分、可信：**

- **标注来源**：补充内容统一用 `> [!tip] 补充` callout，或行内打 `（补：……）`，
  **和原文内容明确分开**，让 Sunny 一眼知道哪些是原文、哪些是 agent 加的。
- **只补成熟可靠的知识**：教科书级、共识性的内容才补；前沿 / 有争议 / 记不准的，
  **标存疑**（`（补，存疑：……）`）或干脆不补，**绝不编造公式和数字**。
- **不喧宾夺主**：补充是补缺口，不是另写一篇。
- **不改写原文结论**：补充与原文冲突时，不直接覆盖，而是并列指出
  （`> [!warning] 此处与原文似有出入：……`）。

示例：
```markdown
**雷诺数 Re**
（补）原文直接用了 Re 未定义。

> [!tip] 补充
> 定义：$Re = \dfrac{\rho v L}{\mu}$，惯性力与粘性力之比。
> 物理意义：Re 小 → 粘性主导、层流；Re 大 → 惯性主导、湍流。
> 工程经验：管内流动临界 $Re \approx 2300$。
```

## 14. 几条铁律（速查）

- **分级叙事是硬骨架**：`一·→1.→（1）→-` 逐级不跳级、同级并列、符号统一。
- ⚠️**缩进每级 2 空格、首级 <4**——≥4 空格起首会被当代码块，`$$` 公式直接不
  渲染。组内不要插空行。
- **逐部分闭环，别图快**：整完一节先自检再下一节。
- **理工科讲透，别压缩成一行**：定义、推导、核心理解三者缺一不可；达不到字数下
  限就是没展开够。
- **推导不跳步**：公式要让读者照着复算，关键步骤标依据。
- **数字 / 公式一个都不能丢**：精读笔记里数字与推导 > 形容词。
- **保留批判视角**：不是摘要，是带判断的精读——方法的毛病、公式的成立条件要写出。
- **简洁汇报**：每部分完成后一句话带过，内容留文件里，别在对话里复述。
- **可补充，但要标注**：用自身知识补全原文缺口时，`> [!tip] 补充` 区分开，只补可
  靠知识，不编造。
- **省 token 不掉质量**：砍重复读取 / 复述 / 回扫，留定义 / 推导 / 核心理解。
- 不确定原文意思时，**标注存疑**而不是编造。
